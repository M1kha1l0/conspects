Репозиторий - директория на локальном/удаленном сервере, в которой хранится код и прочие необходимые файлы проекта.
## Локальное хранение версий
Всем известное сохранение полных копий проекта в отдельной директории под разными названиями версий. Это тоже считается системой контроля версий, вопрос только в ее удобстве.

## SVC (Sub Version Control)
2 поколение системы контроля.
Уже более современная и эффективная система контроля версий для корпоративного использования. Развернута в локальной сети компании.
Есть администратор, который выделяет доступную рабочую область для нового сотрудника. Каждый залив в конце дня читается администратором и вносится в проект с решением конфликтов версий.

Состав команды: Работники, Тимлид, Администратор.

SVC жива до сих пор по нескольким причинам:
- Безопасность.
		Из-за работы в локальной сети доступ к проекту извне невозможен.
- Работник не имеет полную видимость проекта.
		В данной ситуации нельзя испортить весь проект.
- Простота контроля
		Тимлиду проще контролировать проект за счет выделенных участков доступа.
**НО**:
- Невозможно вносить изменения в репозиторий удаленно
		Безопасность это, конечно, хорошо, но времена ковида показали, что нужен баланс.
- Огромная нагрузка на Админа.
		Сохранять изменения в репозитории необходимо каждый день, но каждое новое изменения считается новой версией проекта, поэтому админ должен каждый день решать огромное количество конфликтов при слиянии версий.
- Локальная версия хранится только на сервере в компании.
		Локальное хранение и плохо и хорошо. С одной стороны это безопасность, а с другой риск потерять все, если, например, сервер компании сгорит.
## Git
3 поколение системы контроля.
Современная система контроля с открытым исходным кодом.
Работа **git** основана на сохранении изменений, а не полного списка файлов. Да, когда заливается код в пустой репозиторий, то он полностью хранится на сервере.
Для внесение изменений в репозиторий необходима авторизация, чтобы определять работника, который загрузил говнокод).

Состав:  Работники, Тимлид.

Очевидные плюсы:
- Открытость репозитория.
		Работник видит полную картину проекта и какой именно вклад он вносит. + мотивация.
- Репозиторий хранится глобально.
		Версии репозитория могут храниться как на удаленном сервере, так и в локальной директории.
- Упрощенное управление.
		Как такового администратора нет. Люди могут вносить изменения в репозиторий из любого места, где есть интернет. Конфликты версий решаются самим пользователем.
НО:
- Безопасность.
		Может заливаться вредоносный код для проекта.

## Работа с системой контроля версий
#### Начало работы
Когда мы начинаем работать с удаленны репозиторием **git**, нам необходимо авторизоваться в системе, чтобы вносить изменения. Если же не авторизоваться, то перед каждым внесением изменений нужно  будет вводить  свой логин и пароль. Чтобы таким не заниматься, у гита есть специальные ключи авторизации - **ssh ключи**. чтобы ими пользоваться их надо  сначала сгенерировать, а потом загрузить на гит через сайт.
`ssh-keygen rsa` - bash команда, которая сгенерирует 2 ключа: публичный и приватный в директории **.ssh** на устройстве.  Публичный ключ (он будет с расширением **.pub**) нужно загружать на гит, а приватный никому не показывать.
Для работы также необходимо изучить [[Команды git]].
#### Ветки
Отдельные ветки используются для тестирования нового функционала с сомнительной полезностью, чтобы не вредить основной ветке развития проекта. Если в итоге функционал бесполезен, можно просто удалить ветку и вернуться к исходной версии проекта.

### Паттерны
#### git flow
Над проектом обычно работает несколько команд, которые разрабатывают разный функционал. Когда работает несколько команд, не очень удобно вносить все изменения в одну ветку, поскольку будет очень много конфликтов версий, которые все нужно решать. Поэтому принято решение, что у каждой команды есть отдельная ветка, где ведется разработка той или иной функции.
В каждом репозитории есть ветка **main**, в которой хранится основное тело проекта. В этой ветке обычно лежит продукт с готовым и тестированным функционалом.
Когда команда реализовала функционал, она сливает свою ветку с тестируемой веткой **dev**. Там тестируется новый функционал и только когда все функции отлажены, **dev** сливается с **main**.
#### Другие паттерны
Зависит от команды. У разных команд разные правила ведения репозитория.
